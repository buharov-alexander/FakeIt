## 1. Общее описание проекта

### 1.1 Цель проекта

Разработать веб-версию многопользовательской игры FakeIt — игры на блеф, в которой игроки придумывают ложные ответы на вопросы с целью обмануть других участников, а затем голосуют за правильный ответ.

### 1.2 Целевая аудитория

Компании друзей (2–8 человек), играющие совместно в реальном времени через браузер.

### 1.3 Границы первого этапа (MVP)

Первый этап охватывает только веб-клиент и серверную часть для проведения игровых сессий. Мобильные приложения, аутентификация через сторонние сервисы и монетизация в MVP не предусмотрены.

## 2. Механика игры

### 2.1 Описание игрового процесса

FakeIt — это многопользовательская игра на блеф, основанная на следующем цикле:

1. Ведущий (хост) создаёт игровую комнату и получает код для приглашения.
2. Игроки присоединяются по коду и вводят никнейм.
3. Система показывает всем игрокам один и тот же вопрос с пропуском (факт с пробелом).
4. Каждый игрок вводит свой вымышленный ответ — «ложь».
5. Система собирает все ответы и добавляет правильный ответ.
6. Игроки голосуют за ответ, который считают правильным.
7. Начисляются очки: за обман других игроков и за угаданный правильный ответ.
8. После нескольких раундов подводятся итоги и объявляется победитель.

### 2.2 Система очков

| Событие | Очки |
|---|---|
| Другой игрок проголосовал за твою ложь | +500 |
| Угадал правильный ответ | +1000 |
| Все игроки угадали правильный ответ (без обманутых) | +0 (специальный случай) |

### 2.3 Игровые параметры

- Минимальное количество игроков: 2
- Максимальное количество игроков: 8
- Количество раундов: настраивается хостом (по умолчанию 5)
- Таймер на ввод ответа: настраивается хостом (по умолчанию 90 секунд)
- Таймер на голосование: настраивается хостом (по умолчанию 60 секунд)

## 3. Функциональные требования

### 3.1 Управление комнатой

| ID | Требование |
|---|---|
| FR-01 | Хост может создать игровую комнату и получить уникальный код (6 символов) |
| FR-02 | Игрок может присоединиться к комнате по коду, введя никнейм |
| FR-03 | Хост может удалить игрока из лобби |
| FR-04 | Хост может запустить игру, если в комнате не менее 2 игроков |
| FR-05 | При отключении хоста права передаются следующему игроку |
| FR-06 | При отключении игрока в середине игры — игра продолжается без него |

### 3.2 Игровой процесс

| ID | Требование |
|---|---|
| FR-10 | Система отображает вопрос с пропуском всем игрокам одновременно |
| FR-11 | Каждый игрок вводит один ответ в отведённое время |
| FR-12 | Система отклоняет ответ, совпадающий с правильным (без учёта регистра и пробелов) |
| FR-13 | Система отображает все варианты ответов (вперемешку) для голосования |
| FR-14 | Игрок не может проголосовать за собственный ответ |
| FR-15 | По истечении таймера незаполненные ответы помечаются как «нет ответа» |
| FR-16 | После голосования система раскрывает, кто написал какой ответ |
| FR-17 | Система отображает таблицу очков после каждого раунда |
| FR-18 | По окончании последнего раунда отображается итоговый экран победителя |

### 3.3 База вопросов

| ID | Требование |
|---|---|
| FR-20 | Система содержит встроенную базу русскоязычных вопросов (не менее 100 штук для MVP), хранящихся в файле `questions.json` |
| FR-21 | Вопросы не повторяются в рамках одной игровой сессии |
| FR-22 | Хост может выбрать категорию вопросов (опционально в MVP) |

### 3.4 Интерфейс и UX

| ID | Требование |
|---|---|
| FR-30 | Главный экран: кнопки «Создать игру» и «Присоединиться» |
| FR-31 | Экран лобби отображает список игроков и код комнаты |
| FR-32 | Игровой экран отображает вопрос, таймер и поле для ввода |
| FR-33 | Экран голосования отображает все варианты ответов |
| FR-34 | Экран результатов раунда с анимацией раскрытия авторов ответов |
| FR-35 | Итоговый экран с рейтингом игроков |

## 4. Нефункциональные требования

### 4.1 Производительность

- Задержка отклика интерфейса: не более 1 секунды для пользовательских действий.
- Задержка доставки сообщений через socket.io между игроками: не более 500 мс.
- Время загрузки первого экрана (FCP): не более 2 секунд на среднем интернет-соединении.
- Система должна поддерживать одновременно не менее **50 активных игровых комнат** (до 400 одновременных WebSocket-соединений) на одном сервере без деградации производительности.

### 4.2 Надёжность

- Автоматическое переподключение клиента при потере WebSocket-соединения.
- Сохранение игрового состояния на сервере — при переподключении игрок возвращается в текущий раунд.

### 4.3 Совместимость и адаптивность

- Поддержка актуальных версий браузеров: Chrome, Firefox, Safari, Edge.
- Адаптивная вёрстка: полноценная работа на экранах от 375px (мобильный) до 1920px (десктоп).
- Сенсорный ввод: все интерактивные элементы должны быть удобны для нажатия пальцем (min 44×44px).

### 4.4 Безопасность

- Валидация всех входящих данных на стороне сервера.
- Ограничение частоты запросов (rate limiting) для предотвращения спама.
- Игровые комнаты недоступны без знания кода.

## 5. Архитектура системы

### 5.1 Стек технологий

| Компонент | Технология |
|---|---|
| Фронтенд | Next.js (React), TypeScript |
| Стилизация | Tailwind CSS |
| Реальное время | socket.io |
| Бэкенд | Node.js + Express (отдельный сервер с socket.io) |
| Хранилище состояния игры | In-memory (Map в Node.js процессе) |
| База вопросов | JSON-файл (`public/questions.json`) |
| Язык вопросов | Русский |
| Хостинг (MVP) | Railway (поддерживает persistent WebSocket) |

### 5.2 Высокоуровневая архитектура

```
[Браузер игрока]
      |
      |  HTTP (страницы, статика)       → Next.js (Vercel/Railway)
      |  socket.io (игровые события)    → Express + socket.io сервер
      v
[Express + socket.io сервер]
      |
      |-- Игровая логика (Game Engine)
      |-- Хранилище комнат (in-memory Map)
      |-- База вопросов (questions.json, загружается при старте)
```

### 5.3 Игровые события WebSocket

| Событие | Направление | Описание |
|---|---|---|
| `room:join` | Client → Server | Игрок присоединяется к комнате |
| `room:update` | Server → All | Обновление списка игроков в лобби |
| `game:start` | Server → All | Игра началась, переход к первому раунду |
| `round:question` | Server → All | Новый вопрос раунда |
| `round:answer_submit` | Client → Server | Игрок отправляет свой ответ |
| `round:voting_start` | Server → All | Начало голосования, список вариантов |
| `round:vote_submit` | Client → Server | Игрок голосует за вариант |
| `round:results` | Server → All | Результаты раунда с раскрытием авторов |
| `game:end` | Server → All | Итоговые результаты игры |
| `player:disconnect` | Server → All | Игрок отключился |

## 7. Модели данных

### 7.1 Комната (Room)

```typescript
interface Room {
  code: string;           // Уникальный код комнаты (6 символов)
  hostId: string;         // ID хоста
  players: Player[];      // Список игроков
  status: 'lobby' | 'playing' | 'finished';
  settings: GameSettings;
  gameState?: GameState;
}
```

### 7.2 Игрок (Player)

```typescript
interface Player {
  id: string;             // Socket ID
  nickname: string;
  score: number;
  isConnected: boolean;
}
```

### 7.3 Настройки игры (GameSettings)

```typescript
interface GameSettings {
  maxPlayers: number;       // 2–8
  roundCount: number;       // Количество раундов
  answerTimerSec: number;   // Таймер на ответ
  voteTimerSec: number;     // Таймер на голосование
}
```

### 7.4 Состояние игры (GameState)

```typescript
interface GameState {
  currentRound: number;
  question: Question;
  phase: 'answering' | 'voting' | 'reveal';
  answers: Answer[];        // Ответы игроков + правильный
  votes: Vote[];
}
```

## 8. Этапы разработки (Roadmap)

### Этап 1 — MVP (первый релиз)

- Создание и присоединение к комнате
- Полный игровой цикл (вопрос → ответ → голосование → результаты)
- WebSocket синхронизация в реальном времени
- Встроенная база вопросов (100+ штук)
- Адаптивный интерфейс
- Базовая обработка отключений

### Этап 2 — Улучшения

- Категории вопросов
- Кастомные вопросы (хост добавляет свои)
- Звуковые эффекты и анимации
- Сохранение истории игр

### Этап 3 — Расширение

- Аккаунты и профили игроков
- Глобальная таблица лидеров
- Возможность добавлять вопросы через UI (пользовательский контент)
